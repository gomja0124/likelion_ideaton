# CourtLens 프론트엔드/백엔드/디자인 정리안

## 문서 목적

이 문서는 CourtLens 서비스 기획안을 실제 프로토타입과 개발 범위로 옮기기 위한 실행 정리안이다.

서비스 기획서는 “왜 이 서비스를 만드는가”와 “어떤 사용자 문제를 해결하는가”를 설명한다. 이 문서는 그 기획을 기준으로 프론트엔드, 백엔드, 디자인이 각각 무엇을 구현하고 어떤 범위까지 MVP로 보여줄지 정의한다.

모델 개발은 별도 개인 작업으로 분리한다. 이 문서에서는 모델을 직접 구현하지 않고, 모델이 들어올 수 있는 입력/출력 구조와 서비스 연결 지점만 정의한다.

## 제품 구조 요약

CourtLens는 크게 두 개의 제품 표면으로 구성된다.

| 구분 | 대상 | 목적 | MVP 포함 여부 |
| --- | --- | --- | --- |
| 팬 앱 | 농구 팬 | 경기 전 직관메이트, 경기 후 하이라이트/입덕패스 제공 | 포함 |
| 구단 대시보드 | 구단 마케팅/콘텐츠 담당자 | 팬 반응 분석과 수익화 액션 추천 | 포함 |
| 모델 파이프라인 | 개발자/운영자 | play-by-play와 영상 클립 연결 | 구조만 포함 |
| 관리자 도구 | 서비스 운영자 | 데이터 검수, 권한 관리, 콘텐츠 관리 | MVP 제외 |

## MVP 우선순위

### P0: 반드시 보여줘야 하는 것

- 팬 앱 3개 탭: 하이라이트, 입덕패스, 직관메이트
- 경기 전/경기 후 기능 구분
- 하이라이트 탭 안의 경기 후 하이라이트 메이커
- 입덕패스의 쉬운 해설과 댓글
- 직관메이트 모집글 작성 흐름
- 구단 대시보드의 Fan Insight, Revenue Playbook, Performance
- 샘플 play-by-play 데이터 기반 하이라이트 후보 표시

### P1: 있으면 설득력이 커지는 것

- 선수 클릭 시 선수 정보 패널
- play-by-play 이벤트별 하이라이트 후보 점수 표시
- 구단 대시보드에서 이벤트 초안 생성
- 팬 댓글을 Fan Voice로 요약
- CTA/굿즈/이벤트 성과 예시 데이터

### P2: MVP 이후

- 실제 KBL API 연동
- 실제 영상 URL과 시작/종료 시간 자동 매칭
- 로그인/권한 시스템
- 실제 예매/굿즈몰 연동
- 구단별 커스텀 대시보드
- 다종목 확장

## 프론트엔드 기획

### 프론트엔드 목표

팬 앱은 모바일 앱처럼 보여야 한다. 사용자가 첫 화면에서 “경기 전에는 직관메이트, 경기 후에는 하이라이트”라는 구조를 바로 이해해야 한다.

구단 대시보드는 운영 도구처럼 보여야 한다. 화려한 팬 앱과 달리, 구단 담당자가 빠르게 반응을 확인하고 실행 액션을 고를 수 있어야 한다.

### 현재 MVP 기술 방향

현재 프로토타입은 정적 웹 기반으로 유지한다.

- HTML
- CSS
- Vanilla JavaScript
- 정적 샘플 데이터
- 브라우저에서 바로 확인 가능한 형태

이유:

- 아이디어톤에서는 실제 서비스 완성보다 사용 흐름 설득이 중요하다.
- 팀원이 기능을 빠르게 확인하고 수정하기 쉽다.
- 발표 자료에 넣을 화면 캡처를 만들기 쉽다.

### 추후 제품화 기술 방향

제품화 단계에서는 다음 구조로 전환할 수 있다.

- React 또는 Next.js
- TypeScript
- API 기반 데이터 로딩
- 상태 관리: React Query 또는 Zustand
- 디자인 시스템: 공통 컴포넌트와 팀 컬러 토큰
- 배포: Vercel, Cloudflare Pages, 또는 구단 내부 배포 환경

## 팬 앱 화면 구조

### 1. 하이라이트 탭

경기 후에 열리는 핵심 화면이다.

필수 요소:

- 경기 선택
- 경기 후 상태 표시
- play-by-play 기반 핵심 장면 리스트
- 이 경기 입덕 포인트 선수
- 선수 선택
- 하이라이트 생성 버튼
- 생성된 클립 리스트
- 작전판AI 상세 설명으로 이동

사용자에게 보여야 하는 메시지:

- 이 경기는 이미 끝난 경기다.
- 공식 경기 기록을 기준으로 중요한 장면을 찾는다.
- 팬은 선수를 고르면 그 선수 중심 하이라이트를 받을 수 있다.
- 생성된 하이라이트는 입덕패스와 다음 경기 직관메이트로 이어진다.

### 2. 입덕패스 탭

경기를 잘 모르는 팬이 하이라이트를 이해하는 화면이다.

필수 요소:

- 경기 요약
- 선수별 입덕 포인트
- 장면별 쉬운 설명
- 팬 댓글 작성
- 댓글 기반 Fan Voice 수집 안내

사용자에게 보여야 하는 메시지:

- 농구를 잘 몰라도 이 장면이 왜 중요한지 알 수 있다.
- 댓글은 단순 커뮤니티가 아니라 구단에게 전달되는 팬의 목소리다.

### 3. 직관메이트 탭

경기 전에 열리는 팬 매칭 화면이다.

필수 요소:

- 다음 경기 정보
- 직관 날짜 선택
- 좌석 위치 선택
- 관심 선수 선택
- 모집글 텍스트 작성
- AI 모집글 초안 생성
- 모집글 리스트

사용자에게 보여야 하는 메시지:

- 혼자 직관하기 부담스러운 팬을 연결한다.
- 같은 선수나 같은 장면에 반응한 팬끼리 만날 수 있다.
- 경기 후 하이라이트 반응이 다음 경기 직관으로 이어진다.

## 구단 대시보드 화면 구조

### 1. Fan Insight

팬 반응을 선수, 장면, 행동 단위로 보여준다.

필수 요소:

- 선수별 저장/공유/댓글/클릭 수
- 입덕 포인트 선수
- 팬 반응이 높은 하이라이트
- 직관메이트 클릭 흐름

### 2. Fan Voice

팬 댓글과 불편 사항을 주제별로 정리한다.

필수 요소:

- 선수 콘텐츠 요청
- 좌석/경기장 경험 관련 의견
- 굿즈 요청
- 이벤트 선호
- 초보 팬 질문

### 3. Revenue Playbook

구단이 실행할 수 있는 수익화 액션을 추천한다.

필수 요소:

- 단기 이벤트
- 라운드 이벤트
- 장기 프랜차이즈 스타 마케팅
- 굿즈 테스트
- 스폰서 패키지 후보

### 4. Performance

실행 액션의 성과를 보여준다.

필수 요소:

- 티켓 CTA 클릭
- 굿즈 CTA 클릭
- 이벤트 참여
- 스폰서 이벤트 참여
- 예상 매출 기여

## 백엔드 기획

### 백엔드 목표

백엔드는 팬 앱과 구단 대시보드가 같은 데이터를 바라보도록 만드는 역할을 한다.

팬 앱에서 발생한 행동은 단순 로그가 아니라 구단 대시보드의 인사이트와 수익화 액션으로 이어져야 한다.

### MVP 백엔드 범위

MVP에서는 실제 서버를 반드시 만들 필요는 없다. 다만 백엔드가 있다고 가정했을 때 어떤 데이터와 API가 필요한지 정의해야 한다.

MVP에서는 다음 방식 중 하나를 선택한다.

| 방식 | 설명 | 추천도 |
| --- | --- | --- |
| 정적 JSON | 프론트에서 샘플 JSON을 직접 로드 | 가장 현실적 |
| Mock API | Express/FastAPI로 간단한 API 제공 | 시간이 있으면 가능 |
| 실제 DB | Supabase/Firebase/PostgreSQL 사용 | 아이디어톤 MVP에는 과함 |

현재 단계에서는 정적 JSON 또는 JS 객체 기반 샘플 데이터로 충분하다.

## API 설계 초안

| Method | Endpoint | 설명 |

| GET | /api/teams | 구단 목록 |
| GET | /api/teams/:teamId/players | 구단 선수 목록 |

| GET | /api/clubs/:teamId/insights | 구단 대시보드 인사이트 |

## 모델 연동 지점

모델 개발은 별도 범위지만, 프론트/백엔드에서는 다음 입력과 출력을 받을 수 있게 설계한다.

### 모델 입력

- 경기 ID
- play-by-play 이벤트 목록
- 선수 정보
- 경기 스코어 흐름
- 연결 가능한 영상 URL
- 팬 반응 데이터

### 모델 출력

- 하이라이트 후보 이벤트
- 선수별 영향도 점수
- 추천 선수
- 클립 제목
- 초보자용 설명
- 구단 활용 액션 후보

프론트엔드는 이 출력을 화면에 표시하고, 백엔드는 이 출력을 저장하거나 구단 대시보드 인사이트로 변환한다.

## 디자인 기획

### 디자인 목표

CourtLens 디자인은 팬 앱과 구단 대시보드의 목적이 다르다는 점을 반영해야 한다.

팬 앱은 쉽고 직관적이어야 한다. 초보 팬이 부담 없이 들어와서 “무엇을 보면 되는지” 바로 알 수 있어야 한다.

구단 대시보드는 조용하고 실무적이어야 한다. 구단 담당자가 팬 반응, 추천 액션, 예상 성과를 빠르게 비교할 수 있어야 한다.

## 디자인 원칙

### 팬 앱

- 모바일 우선
- 카드 기반 정보 구조
- 경기 전/경기 후 상태를 명확히 표시
- 어려운 농구 용어를 짧게 설명
- 선수와 장면을 중심으로 감정적 연결 만들기
- 버튼은 다음 행동이 명확해야 함

### 구단 대시보드

- 데이터 스캔이 쉬운 구조
- 팬 반응과 수익화 액션을 한 화면에서 연결
- 과한 장식보다 명확한 정보 밀도 우선
- 각 카드가 클릭 가능한 세부 분석으로 이어져야 함
- 이벤트/스폰서/성과를 실행 단위로 보여줘야 함

## 디자인 시스템 초안

### 컬러

- 기본 배경: 따뜻한 오프화이트
- 강조색: 선택 구단의 팀 컬러
- 보조 강조: 하이라이트/이벤트용 노란색 계열
- 구단 대시보드: 팬 앱보다 채도를 낮추고 데이터 가독성 우선

### 타이포그래피

- 제목: 굵고 짧게
- 설명: 1~2문장 이하
- 카드 내부 텍스트: 작지만 가독성 있게
- 버튼 텍스트: 동사 중심

### 컴포넌트

- 탭
- 경기 상태 배지
- 선수 칩
- 하이라이트 카드
- play-by-play 이벤트 행
- 입덕패스 설명 카드
- 댓글 카드
- 직관메이트 모집 카드
- 구단 KPI 카드
- Revenue Playbook 카드
- Event Builder 모달

## 구현 리스크

### 1. play-by-play와 영상 시간 매칭

경기 시계 시간과 실제 영상 시간은 다르다. MVP에서는 자동 매칭을 완성하지 않고, 수동 연결 또는 샘플 연결로 보여준다.

### 2. 기능 범위 과다

팬 앱, 구단 대시보드, 모델을 모두 완성하려 하면 범위가 커진다. MVP에서는 서비스 흐름을 보여주는 데 집중한다.

### 3. B2B 수익화 설득

구단이 비용을 낼 이유가 분명해야 한다. 단순 팬 앱이 아니라 팬 반응을 이벤트, 굿즈, 스폰서십으로 전환하는 대시보드가 핵심이다.

### 4. 디자인 복잡도

농구 정보가 많기 때문에 화면이 복잡해질 수 있다. 초보 팬 화면은 정보량을 줄이고, 구단 화면은 데이터 밀도를 높이는 식으로 분리한다.

## 최종 정리

CourtLens MVP의 개발 초점은 완전한 AI 모델이 아니라, 팬 앱과 구단 대시보드가 하나의 데이터 흐름으로 연결되는 모습을 보여주는 것이다.

프론트엔드는 경기 전/후 팬 경험을 명확히 보여주고, 백엔드는 경기/선수/play-by-play/팬 반응 데이터 구조를 정의하며, 디자인은 팬 앱의 쉬운 이해와 구단 대시보드의 실행 가능성을 동시에 전달해야 한다.
